# 챕터 13: 커스텀 슬래시 커맨드와 스킬

---

## 13.1 왜 커스터마이즈하는가

여러분은 매일 같은 명령을 반복합니다.
PR 리뷰, 배포, 테스트 실행, 로그 분석.
매번 같은 프롬프트를 타이핑하고 있다면,
그것은 자동화의 신호입니다.

커스텀 슬래시 커맨드는 이 문제를 해결합니다.
자주 쓰는 워크플로우를 한 번 정의하면,
`/deploy`처럼 한 단어로 실행할 수 있습니다.

> **팁**: 3번 이상 반복한 작업이 있다면, 스킬로 만드세요. 투자 대비 효과가 큽니다.

스킬의 핵심 장점을 정리합니다.

| 장점 | 설명 |
|------|------|
| 일관성 | 매번 동일한 절차를 실행합니다 |
| 속도 | 긴 프롬프트를 한 단어로 축약합니다 |
| 공유 | 팀원 전체가 같은 워크플로우를 씁니다 |
| 컨텍스트 절약 | 메인 대화 안에서 실행되어 효율적입니다 |

---

## 13.2 스킬 만들기

스킬은 `.claude/skills/` 디렉토리에 만듭니다.
디렉토리 이름이 곧 커맨드 이름이 됩니다.
각 디렉토리 안에 `SKILL.md` 파일을 둡니다.

### 디렉토리 구조

```
프로젝트/
└── .claude/
    └── skills/
        └── deploy/
            └── SKILL.md
```

### SKILL.md 기본 형식

`SKILL.md`는 YAML frontmatter로 시작합니다.
그 아래에 프롬프트 본문을 작성합니다.

```yaml
---
name: deploy
description: 프로덕션 환경에 배포합니다
---
다음 단계를 순서대로 실행하세요:

1. 테스트를 실행합니다
2. 빌드를 생성합니다
3. 배포 스크립트를 실행합니다
```

`name`은 커맨드 표시 이름입니다.
`description`은 자동 완성 시 보이는 설명입니다.
이 두 필드는 필수입니다.

스킬을 만든 뒤 `/deploy`를 입력하면,
Claude가 SKILL.md의 지시를 따릅니다.

> **주의**: `SKILL.md` 파일명은 정확히 대문자여야 합니다. `skill.md`로 쓰면 인식하지 못합니다.

## 13.3 동적 컨텍스트: 실행 결과 주입

스킬의 강력한 기능 중 하나입니다.
`!` 뒤에 백틱으로 감싼 명령어를 쓰면,
실행 결과가 프롬프트에 자동 주입됩니다.

### 기본 문법

```yaml
---
name: review-pr
description: 현재 PR을 리뷰합니다
---
아래 PR diff를 리뷰해주세요.

PR diff:
!`gh pr diff`

현재 브랜치:
!`git branch --show-current`
```

`/review-pr`을 실행하면 다음이 일어납니다.

1. `gh pr diff` 명령이 먼저 실행됩니다
2. 그 출력이 프롬프트에 삽입됩니다
3. `git branch` 결과도 함께 삽입됩니다
4. 완성된 프롬프트가 Claude에 전달됩니다

이 방식으로 실시간 데이터를 주입할 수 있습니다.
Git 상태, API 응답, 로그 등 무엇이든 됩니다.

> **팁**: 동적 컨텍스트는 스킬 호출 시점에 실행됩니다. 매번 최신 결과를 가져옵니다.

## 13.4 인자와 옵션

### $ARGUMENTS로 인자 받기

사용자가 입력한 추가 텍스트를 받을 수 있습니다.
`$ARGUMENTS`로 전체 인자를 참조합니다.
`$0`, `$1`처럼 개별 인자도 사용 가능합니다.

```yaml
---
name: test
description: 특정 파일의 테스트를 실행합니다
argument-hint: [파일경로]
---
다음 파일의 테스트를 작성하고 실행하세요:
$ARGUMENTS
```

사용 예시는 다음과 같습니다.

```
> /test src/utils/parser.ts
```

`argument-hint`는 자동 완성에 힌트를 줍니다.
사용자가 어떤 인자를 넣을지 안내합니다.

### allowed-tools: 도구 제한

스킬이 사용할 도구(Tool)를 제한할 수 있습니다.
불필요한 도구 접근을 차단하여 안전합니다.

```yaml
---
name: analyze
description: 코드를 분석합니다 (읽기 전용)
allowed-tools: Read, Grep, Glob
---
다음 코드베이스를 분석해주세요.
파일을 수정하지 마세요.
```

### 기타 고급 옵션

| 옵션 | 설명 | 예시 |
|------|------|------|
| `model` | 실행 모델을 지정합니다 | `model: haiku` |
| `context` | 컨텍스트 처리 방식입니다 | `context: fork` |
| `agent` | 특정 에이전트로 위임합니다 | `agent: Explore` |
| `disable-model-invocation` | 자동 호출을 비활성화합니다 | `true` |

`context: fork`를 쓰면 별도 컨텍스트에서 실행됩니다.
메인 대화의 컨텍스트를 오염시키지 않습니다.

`model: haiku`를 쓰면 빠르고 저렴하게 실행됩니다.
간단한 작업에 적합합니다.

## 13.5 팀 공유: 프로젝트 vs 사용자 범위

스킬에는 두 가지 저장 위치가 있습니다.
용도에 따라 적절히 선택하세요.

### 프로젝트 범위

```
프로젝트/.claude/skills/deploy/SKILL.md
```

Git에 커밋하여 팀 전체가 공유합니다.
프로젝트 특화 워크플로우에 적합합니다.
PR을 통해 리뷰하고 개선할 수 있습니다.

### 사용자 범위

```
~/.claude/skills/my-util/SKILL.md
```

개인 홈 디렉토리에 저장합니다.
모든 프로젝트에서 사용할 수 있습니다.
개인 취향의 유틸리티에 적합합니다.

### 우선순위

같은 이름의 스킬이 양쪽에 있을 때,
프로젝트 범위가 우선 적용됩니다.
팀 표준을 개인 설정보다 우선합니다.

> **팁**: 팀 공통 워크플로우는 프로젝트 범위에, 개인 단축키는 사용자 범위에 두세요. `.claude/skills/`를 `.gitignore`에 넣지 마세요.

## 13.6 실습: 배포 스킬 만들기

실제로 배포 스킬을 만들어 봅니다.
단계별로 따라 하세요.

### 1단계: 디렉토리 생성

```bash
mkdir -p .claude/skills/deploy
```

### 2단계: SKILL.md 작성

```yaml
---
name: deploy
description: 스테이징 또는 프로덕션에 배포합니다
argument-hint: [staging|production]
allowed-tools: Read, Bash, Grep
---
## 배포 워크플로우

배포 대상 환경: $ARGUMENTS (기본값: staging)

현재 상태:
!`git status --short`

최근 커밋:
!`git log --oneline -5`

다음 절차를 순서대로 수행하세요:

1. `npm test`로 테스트를 실행합니다
2. 테스트 실패 시 즉시 중단합니다
3. `npm run build`로 빌드합니다
4. 환경에 맞는 배포 스크립트를 실행합니다
   - staging: `./scripts/deploy-staging.sh`
   - production: `./scripts/deploy-prod.sh`
5. 배포 결과를 요약합니다
```

### 3단계: 실행

```
> /deploy staging
```

이 한 줄로 전체 배포 절차가 시작됩니다.
Git 상태와 최근 커밋이 자동으로 주입됩니다.
테스트, 빌드, 배포가 순서대로 진행됩니다.

### 4단계: Git 커밋으로 팀 공유

```bash
git add .claude/skills/deploy/SKILL.md
git commit -m "배포 스킬 추가"
```

이제 팀원 누구나 `/deploy`를 사용합니다.

> **주의**: `allowed-tools`에 `Bash`를 포함하면 셸 명령이 실행됩니다. 프로덕션 배포 스킬에는 반드시 실행 전 확인 절차를 프롬프트에 포함하세요.

## 정리

이번 챕터에서 배운 내용을 정리합니다.

| 개념 | 핵심 |
|------|------|
| 스킬 위치 | `.claude/skills/이름/SKILL.md` |
| 필수 필드 | `name`, `description` |
| 동적 주입 | `!` + 백틱 명령어 |
| 인자 처리 | `$ARGUMENTS`, `$0`, `$1` |
| 도구 제한 | `allowed-tools` |
| 프로젝트 공유 | `.claude/skills/` (Git 커밋) |
| 개인 사용 | `~/.claude/skills/` |

다음 챕터에서는 훅(Hook) 시스템을 다룹니다.
스킬 실행 전후에 자동 검증을 추가하는 방법,
그리고 보안 정책을 코드로 관리하는 방법을 배웁니다.
