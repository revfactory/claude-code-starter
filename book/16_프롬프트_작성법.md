# 챕터 16: 효과적인 프롬프트 작성법

> **학습 목표**
> - 프롬프트의 3요소를 구조화할 수 있다
> - 나쁜 프롬프트를 좋은 프롬프트로 개선한다
> - 탐색→계획→정제→구현 전략을 실천한다
> - 컨텍스트를 효과적으로 제공할 수 있다
> - 흔한 실수 3가지를 사전에 방지한다

---

## 16.1 프롬프트의 3요소: 목표, 범위, 검증 방법

Claude Code에게 일을 맡기는 것은 동료에게 업무를 위임하는 것과 같습니다. "대충 해줘"라고 말하면 결과도 대충 나옵니다. 좋은 프롬프트에는 반드시 3가지 요소가 있습니다.

### 목표(What)

무엇을 달성할지 명확히 말합니다. "개선해줘"가 아니라 "응답 시간을 50% 줄여줘"입니다.

### 범위(Where)

어디를 건드릴지 한정합니다. 파일, 디렉터리, 함수 이름을 지정합니다. 범위가 넓을수록 결과가 흐려집니다.

### 검증 방법(How to verify)

완료 기준을 제시합니다. 테스트 통과, 타입 체크, 빌드 성공 등 구체적인 확인 방법입니다.

```text
# 3요소를 갖춘 프롬프트 예시

목표: JWT 토큰 갱신 로직을 추가해줘
범위: src/auth/token.ts 파일만 수정
검증: npm test -- --grep "token refresh" 통과
```

> **팁**: 3요소 중 하나라도 빠지면 Claude Code가 추측합니다. 추측은 곧 재작업입니다. 30초 더 투자해서 프롬프트를 완성하세요.

## 16.2 좋은 프롬프트 패턴: Before/After 비교

나쁜 프롬프트와 좋은 프롬프트를 5쌍 비교합니다. 차이를 체감해 보세요.

### 비교 1: 버그 수정

```text
# Before (나쁜 프롬프트)
로그인이 안 돼. 고쳐줘.

# After (좋은 프롬프트)
만료된 카드로 결제 시 로그인 세션이 끊긴다.
src/payments/checkout.ts를 확인해줘.
실패 테스트를 먼저 작성하고 수정해줘.
```

**차이점**: 증상, 위치, 검증 방법이 모두 포함됩니다.

### 비교 2: 리팩토링

```text
# Before
이 코드 좀 깔끔하게 해줘.

# After
src/utils/dateHelper.ts의 중복 함수 3개를
하나의 제네릭 함수로 통합해줘.
기존 테스트가 모두 통과해야 한다.
```

**차이점**: "깔끔"은 주관적입니다. 구체적 행동을 지시합니다.

### 비교 3: 새 기능

```text
# Before
알림 기능을 만들어줘.

# After
사용자가 댓글을 달면 게시글 작성자에게
이메일 알림을 보내는 기능을 추가해줘.
@src/notifications/ 디렉터리에 만들고
기존 EmailService를 재사용해줘.
```

**차이점**: 트리거, 대상, 위치, 의존성이 명시됩니다.

### 비교 4: 성능 개선

```text
# Before
API가 느려. 최적화해줘.

# After
GET /api/products 응답이 3초 걸린다.
N+1 쿼리가 원인으로 의심된다.
src/routes/products.ts에서 쿼리를 확인하고
응답 시간 1초 이내로 개선해줘.
```

**차이점**: 측정 가능한 목표와 가설이 있습니다.

### 비교 5: 코드 리뷰

```text
# Before
이 PR 리뷰해줘.

# After
PR #42를 리뷰해줘. 다음에 집중해줘:
1. SQL 인젝션 취약점
2. 에러 핸들링 누락
3. 타입 안전성
심각도 상/중/하로 분류해줘.
```

**차이점**: 관점과 출력 형식이 지정됩니다.

> **주의**: 프롬프트가 5줄을 넘어가면 과합니다. 핵심 정보만 담으세요. 장황한 설명은 오히려 혼란을 줍니다.

## 16.3 반복 대화 전략: 탐색 → 계획 → 정제 → 구현

한 번의 프롬프트로 완벽한 결과를 기대하지 마세요. 4단계 반복 전략이 효과적입니다.

### 1단계: 탐색

현재 상태를 파악합니다. Claude Code에게 분석을 요청합니다.

```text
src/auth/ 모듈의 구조를 분석해줘.
인증 흐름에서 개선할 점이 있으면 알려줘.
```

이 단계에서는 코드 수정을 요청하지 않습니다. 상황을 이해하는 것이 목적입니다.

### 2단계: 계획

탐색 결과를 바탕으로 계획을 세웁니다.

```text
방금 분석한 내용을 토대로
세션 타임아웃 개선 계획을 세워줘.
변경할 파일 목록과 순서를 정리해줘.
```

> **팁**: 이 단계에서 Plan 모드(`Shift+Tab`)를 활용하면 코드 수정 없이 안전하게 계획을 검토할 수 있습니다.

### 3단계: 정제

계획에서 우선순위를 좁힙니다.

```text
계획 중 1번 "세션 타임아웃"만 먼저 진행하자.
Redis 대신 기존 메모리 스토어를 사용해줘.
```

한 번에 하나의 작업만 지시합니다. 여러 작업을 한꺼번에 맡기면 품질이 떨어집니다.

### 4단계: 구현

정제된 범위에서 실행합니다.

```text
src/auth/session.ts에 타임아웃 로직을 구현해줘.
만료 시간은 환경변수 SESSION_TTL로 설정한다.
테스트를 먼저 작성하고 구현해줘.
```

![반복 대화 전략](images/ch16_iterative_strategy.png)

> **팁**: 각 단계 사이에 Claude Code의 응답을 꼼꼼히 읽으세요. 잘못된 방향을 조기에 수정할 수 있습니다.

## 16.4 컨텍스트 제공: @파일, 이미지, 에러 로그

좋은 프롬프트에는 충분한 컨텍스트가 필요합니다. Claude Code가 지원하는 3가지 방법을 소개합니다.

### @파일 참조

`@` 뒤에 파일 경로를 붙이면 해당 파일을 컨텍스트에 포함합니다.

```text
@src/auth/token.ts 의 refreshToken 함수에서
만료 체크 로직이 빠져 있다.
만료 10분 전 자동 갱신을 추가해줘.
```

디렉터리도 참조할 수 있습니다. `@src/auth/` 형태로 쓰면 해당 폴더의 파일 목록을 전달합니다.

### 이미지 붙여넣기

디자인 시안이나 에러 스크린샷을 직접 붙여넣을 수 있습니다. `Ctrl+V`(macOS는 `Cmd+V`)로 클립보드의 이미지를 전달합니다.

```text
[이미지 붙여넣기]
이 디자인대로 버튼 컴포넌트를 구현해줘.
색상 코드는 이미지에서 추출해줘.
```

스크린샷 한 장이 긴 설명보다 낫습니다. UI 버그를 신고할 때 특히 유용합니다.

### 에러 로그 전달

에러 메시지를 그대로 복사해서 붙여넣으세요. Claude Code가 스택 트레이스를 분석합니다.

```text
다음 에러가 발생한다:

TypeError: Cannot read properties of undefined
  (reading 'accessToken')
  at refreshSession (src/auth/session.ts:42)
  at processTicksAndRejections (node:internal)

원인을 파악하고 수정해줘.
```

> **주의**: 민감 정보가 포함된 로그는 주의하세요. API 키, 비밀번호, 개인정보가 있다면 마스킹 후 전달합니다.

### MCP 서버 활용

MCP 서버가 연결되어 있다면 외부 데이터도 컨텍스트로 활용할 수 있습니다. Jira 이슈, Slack 메시지, DB 스키마 등을 직접 참조합니다.

```text
Jira PROJ-123 이슈를 확인하고
해당 버그를 수정해줘.
```

## 16.5 흔한 실수: Kitchen Sink 세션, 반복 수정, 과도한 위임

프롬프트 작성법을 알아도 실수는 반복됩니다. 가장 흔한 3가지 함정을 짚어봅니다.

### 실수 1: Kitchen Sink 세션

하나의 세션에서 무관한 작업을 섞는 것입니다.

```text
# 이렇게 하지 마세요
인증 버그를 고치고, 대시보드 UI를 바꾸고,
README도 업데이트하고, 배포 스크립트도 수정해줘.
```

서로 다른 작업은 컨텍스트를 오염시킵니다. 인증 코드 수정 중에 대시보드 CSS가 컨텍스트를 차지합니다. Claude Code의 집중력이 분산됩니다.

**해결책**: 작업마다 `/clear`로 세션을 분리하세요. 워크스페이스가 깨끗해야 결과도 깨끗합니다.

### 실수 2: 반복 수정

같은 문제를 2회 이상 수정 요청하는 것입니다.

```text
# 첫 번째 시도
"버튼 색을 파란색으로 바꿔줘"
# 두 번째
"아니, 좀 더 진한 파란색으로"
# 세 번째
"그것도 아니고, #1a73e8로 해줘"
```

매번 수정할 때마다 컨텍스트를 소모합니다. 처음부터 정확한 값을 제시했다면 한 번에 끝났습니다.

**해결책**: 2번 이상 수정하게 되면 멈추세요. 더 정확한 프롬프트를 작성하고 `/clear` 후 새로 시작하세요.

> **팁**: `/compact`로 이전 대화를 압축하는 방법도 있습니다. 하지만 반복 수정이 잦다면 프롬프트 자체를 개선하는 것이 근본 해결입니다.

### 실수 3: 과도한 위임

모든 판단을 Claude Code에게 맡기는 것입니다.

```text
# 이렇게 하지 마세요
이 프로젝트를 전체적으로 개선해줘.
알아서 좋은 방향으로 리팩토링해줘.
```

"알아서"는 위험한 단어입니다. Claude Code는 여러분의 비즈니스 맥락을 모릅니다. 기술적으로 우아해도 비즈니스 요구와 맞지 않을 수 있습니다.

**해결책**: 판단은 여러분이, 실행은 Claude Code가 합니다. 방향을 결정한 뒤 구체적 작업을 위임하세요.

```text
# 좋은 위임
다음 3가지 규칙으로 리팩토링해줘:
1. 함수당 20줄 이내
2. 순환 의존성 제거
3. 기존 테스트 100% 통과 유지
```

### 자가 진단 체크리스트

프롬프트를 보내기 전에 확인하세요.

| 항목 | 확인 |
|------|------|
| 목표가 한 문장으로 정리되는가? | ☐ |
| 수정 범위(파일/함수)가 명시되는가? | ☐ |
| 완료 기준(테스트/빌드)이 있는가? | ☐ |
| 한 세션에 하나의 주제만 다루는가? | ☐ |
| "알아서"라는 표현이 없는가? | ☐ |

## 마무리

프롬프트는 Claude Code와의 계약서입니다. 목표, 범위, 검증 방법이라는 3요소를 갖추면 결과의 품질이 달라집니다. 탐색에서 시작해 점진적으로 구체화하세요. 한 번에 완벽할 필요는 없습니다. 다만 같은 실수를 반복하지 않는 것이 중요합니다.

> **핵심 정리**
> - 프롬프트 3요소: 목표 + 범위 + 검증 방법
> - 4단계 전략: 탐색 → 계획 → 정제 → 구현
> - 컨텍스트: @파일, 이미지, 에러 로그 활용
> - 세션 위생: 한 주제, 한 세션 원칙 준수
> - 판단은 사람이, 실행은 Claude Code가
