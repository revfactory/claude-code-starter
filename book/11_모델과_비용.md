# 챕터 11: 모델 선택과 비용 관리

> **학습 목표**
> - 사용 가능한 모델(Sonnet, Opus, Haiku)의 특성을 안다
> - 작업에 맞는 모델을 선택할 수 있다
> - Effort Level을 조절할 수 있다
> - 비용을 모니터링하고 최적화할 수 있다

---

## 11.1 모델 3형제: Sonnet, Opus, Haiku

Claude Code는 세 가지 모델을 제공합니다. 각각 성격이 뚜렷합니다. 용도에 맞게 골라 쓰는 것이 핵심입니다.

### Sonnet — 일상의 파트너

Sonnet은 기본 모델입니다. 속도와 품질의 균형이 뛰어납니다. 일반적인 코딩 작업 대부분을 소화합니다. 비용도 합리적입니다.

**잘하는 일**: 코드 수정, 버그 수정, 테스트 작성, 리팩토링, 문서화

### Opus — 깊은 사고가 필요할 때

Opus는 최상위 모델입니다. 복잡한 아키텍처 설계에 강합니다. 다중 파일에 걸친 대규모 리팩토링도 훌륭하게 처리합니다. 대신 비용이 높고 속도가 느립니다.

**잘하는 일**: 아키텍처 설계, 대규모 리팩토링, 복잡한 디버깅, 성능 최적화

### Haiku — 빠른 응답이 필요할 때

Haiku는 경량 모델입니다. 응답 속도가 매우 빠릅니다. 단순 질문이나 반복 작업에 적합합니다. 비용이 가장 낮습니다.

**잘하는 일**: 간단한 질문, 코드 설명, 단순 수정, 배경 작업

### 비용 비교

| 모델 | 속도 | 품질 | 상대 비용 | 일일 예상 |
|------|------|------|----------|----------|
| **Sonnet** | 빠름 | 높음 | 중간 | ~$6 |
| **Opus** | 느림 | 최상 | 높음 | ~$15+ |
| **Haiku** | 매우 빠름 | 보통 | 낮음 | ~$1 |

### 모델 별칭

Claude Code는 편리한 별칭을 제공합니다.

| 별칭 | 설명 |
|------|------|
| `default` | 계정에 맞는 추천 모델 |
| `sonnet` | 최신 Sonnet (현재 4.6) |
| `opus` | 최신 Opus (현재 4.6) |
| `haiku` | 최신 Haiku |
| `sonnet[1m]` | 1M 토큰 컨텍스트 Sonnet |
| `opusplan` | Opus가 계획, Sonnet이 실행 |

> **팁**: `opusplan`은 똑똑한 조합입니다. Opus의 사고력으로 계획을 세우고, Sonnet이 실행합니다. 비용 대비 품질이 우수합니다.

## 11.2 언제 어떤 모델을: 작업별 선택 가이드

모델 선택은 작업의 복잡도로 결정합니다. 아래 표를 참고하세요.

| 작업 유형 | 추천 모델 | 이유 |
|----------|----------|------|
| 일반 코드 수정 | Sonnet | 속도와 품질의 균형 |
| 버그 수정 | Sonnet | 충분한 추론 능력 |
| 아키텍처 설계 | Opus | 깊은 사고 필요 |
| 대규모 리팩토링 | Opus / opusplan | 여러 파일 연관 분석 |
| 코드 설명 | Sonnet / Haiku | 단순 해석 작업 |
| 테스트 작성 | Sonnet | 패턴 이해 필요 |
| 단순 질문 | Haiku | 빠른 응답이 핵심 |
| PR 리뷰 | Opus | 맥락 파악이 중요 |
| 반복 자동화 | Haiku | 비용 절약 |
| 복잡한 디버깅 | Opus | 깊은 추론 필요 |

### 모델 변경 방법

세션 중에도 자유롭게 변경할 수 있습니다.

```bash
# 세션 중 변경
/model opus

# 시작할 때 지정
claude --model opus

# 환경 변수로 기본값 설정
export CLAUDE_MODEL=opus
```

> **팁**: 하나의 세션에서 모델을 바꿔가며 사용하세요. 탐색은 Sonnet, 핵심 설계는 Opus, 단순 확인은 Haiku. 이런 전환이 비용을 크게 줄여줍니다.

## 11.3 Effort Level: 응답 깊이 조절

모델뿐 아니라 **응답 깊이**도 조절할 수 있습니다. Effort Level은 모델이 얼마나 깊이 생각할지 결정합니다.

| 레벨 | 특징 | 적합한 상황 |
|------|------|-----------|
| **Low** | 짧고 간결한 응답 | 단순 질문, Yes/No |
| **Medium** | 균형 잡힌 응답 (기본값) | 일반 코딩 작업 |
| **High** | 깊은 분석과 상세 응답 | 복잡한 문제 해결 |

### 조절 방법

```bash
# 세션 중 조절
/model          # 실행 후 화살표 키로 선택

# 환경 변수로 설정
export CLAUDE_CODE_EFFORT_LEVEL=high
```

Effort Level을 낮추면 토큰 사용량이 줄어듭니다. 간단한 작업에 High를 쓰면 비용만 낭비됩니다.

### 조합 전략

모델과 Effort Level을 조합하면 더 세밀하게 조절할 수 있습니다.

| 상황 | 모델 | Effort | 효과 |
|------|------|--------|------|
| 간단한 수정 | Sonnet | Low | 빠르고 저렴 |
| 일반 개발 | Sonnet | Medium | 균형 잡힘 |
| 복잡한 설계 | Opus | High | 최고 품질 |
| 빠른 확인 | Haiku | Low | 최소 비용 |

> **주의**: Effort Level `High`는 Extended Thinking을 활성화합니다. 토큰 소비가 크게 증가합니다. 꼭 필요한 경우에만 사용하세요.

## 11.4 비용 모니터링

비용 관리의 첫걸음은 **현황 파악**입니다. Claude Code는 실시간 비용 확인 기능을 제공합니다.

### /cost 커맨드

세션 중 언제든 비용을 확인할 수 있습니다.

```bash
/cost
```

출력 예시:

```
Session cost: $2.34
  Input tokens:  45,230  ($0.68)
  Output tokens: 12,100  ($1.44)
  Cache read:    8,500   ($0.09)
  Cache write:   3,200   ($0.13)
```

입력 토큰과 출력 토큰의 비용이 다릅니다. 출력 토큰이 더 비쌉니다. 따라서 응답이 길수록 비용이 증가합니다.

### 일일/월간 비용 예상

일반적인 개발자 기준 비용을 정리했습니다.

| 사용 패턴 | 일일 예상 | 월간 예상 |
|----------|----------|----------|
| 가벼운 사용 (Sonnet 위주) | $2~4 | $40~80 |
| 보통 사용 (Sonnet + 가끔 Opus) | $5~8 | $100~160 |
| 집중 사용 (Opus 빈번) | $10~20 | $200~400 |
| Haiku 위주 자동화 | $0.5~2 | $10~40 |

> **팁**: `/cost`를 습관적으로 확인하세요. 특히 Opus 모델을 사용한 뒤에는 반드시 체크합니다. 예상치 못한 비용 폭탄을 방지할 수 있습니다.

### Pro/Max 구독 사용자

Pro/Max 구독자는 사용량 한도 내에서 추가 비용이 없습니다. 다만 한도를 초과하면 속도가 제한됩니다. `/cost`로 남은 한도를 추적하세요.

## 11.5 비용 절약 팁

비용을 아끼면서도 생산성을 유지하는 전략을 소개합니다.

### 1. 컨텍스트 관리

컨텍스트가 커지면 매 요청마다 토큰이 늘어납니다. 주기적으로 정리하세요.

```bash
# 대화 압축 — 핵심만 남깁니다
/compact

# 완전 초기화 — 새 작업 시작 시
/clear
```

세션이 길어질수록 비용이 가속됩니다. 작업 주제가 바뀌면 `/clear`로 초기화하세요.

### 2. Subagent 활용

큰 작업을 작은 단위로 나눠서 위임합니다. 각 Subagent는 독립된 컨텍스트를 사용하므로 토큰 효율이 좋습니다.

```
# 프롬프트 예시
"이 작업을 subagent로 나눠서 처리해줘.
1) auth 모듈 테스트 작성
2) API 라우트 리팩토링
3) README 업데이트"
```

### 3. MCP 서버 최적화

연결된 MCP 서버가 많으면 시스템 프롬프트가 커집니다. 사용하지 않는 MCP 서버는 비활성화하세요.

```bash
# MCP 서버 상태 확인
/mcp

# 불필요한 서버 제거
claude mcp remove unused-server
```

### 4. 추가 절약 전략

| 전략 | 효과 | 실행 방법 |
|------|------|----------|
| Sonnet을 기본으로 | 비용 60% 절감 | `default` 유지 |
| 짧은 세션 유지 | 컨텍스트 비용 절감 | 작업 단위로 `/clear` |
| CLAUDE.md 간소화 | 매 요청 토큰 절감 | 200줄 이하 유지 |
| Skill로 규칙 분리 | CLAUDE.md 축소 | `.claude/skills/` 활용 |
| Extended Thinking 제한 | 출력 토큰 절감 | `MAX_THINKING_TOKENS` 설정 |

> **팁**: 가장 효과적인 절약법은 **모델 선택**입니다. 대부분의 작업은 Sonnet이면 충분합니다. Opus는 정말 복잡한 문제에만 투입하세요.

## 정리

| 개념 | 핵심 요약 |
|------|---------|
| 모델 3형제 | Sonnet(균형), Opus(품질), Haiku(속도) |
| 모델 선택 | 작업 복잡도에 따라 결정 |
| Effort Level | Low/Medium/High로 응답 깊이 조절 |
| 비용 확인 | `/cost`로 실시간 모니터링 |
| 비용 절약 | 컨텍스트 관리 + 모델 선택이 핵심 |

모델 선택은 **도구 선택**과 같습니다. 나사를 조일 때 망치를 쓰지 않듯, 간단한 수정에 Opus를 쓸 필요는 없습니다. 상황에 맞는 모델과 Effort Level을 조합하면, 비용은 줄이면서 생산성은 높일 수 있습니다. 다음 챕터에서는 컨텍스트 관리의 기술을 더 깊이 다루겠습니다.
