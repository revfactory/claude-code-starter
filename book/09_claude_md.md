# 챕터 9: CLAUDE.md — 프로젝트의 두뇌

> **학습 목표**
> - CLAUDE.md의 역할과 로딩 계층을 이해한다
> - 효과적인 CLAUDE.md를 작성할 수 있다
> - `/init`으로 자동 생성하고 수정할 수 있다
> - 너무 커지지 않도록 관리할 수 있다

---

여러분이 새 팀원에게 업무를 맡긴다고 생각해보세요.
"우리 프로젝트는 이렇게 빌드하고,
코드 스타일은 이런 규칙을 따르고,
테스트는 이렇게 돌려야 해."
이런 설명이 필요하겠죠?

Claude Code에게도 똑같은 브리핑이 필요합니다.
그 브리핑 문서가 바로 `CLAUDE.md`입니다.
이번 챕터에서는 프로젝트의 두뇌를
설계하는 방법을 배워보겠습니다.

---

## 9.1 CLAUDE.md란

`CLAUDE.md`는 프로젝트 규칙 파일입니다.
Claude Code가 세션을 시작할 때 자동으로 읽습니다.
별도의 지시 없이, 매번 알아서 로드됩니다.

어떤 내용을 담을까요?

- **빌드 명령어**: `npm run build`, `pytest` 등
- **코드 스타일**: 네이밍 규칙, 들여쓰기, 린트 설정
- **테스트 규칙**: 테스트 프레임워크, 실행 방법
- **프로젝트 구조**: 주요 디렉토리의 역할
- **금지 사항**: "이 파일은 건드리지 마세요"

한마디로, Claude Code가 여러분의 프로젝트에서
지켜야 할 모든 규칙을 모아놓은 곳입니다.

> **팁**: `CLAUDE.md`는 마크다운 형식입니다. 사람이 읽어도 이해하기 쉽게 작성하세요. 좋은 `CLAUDE.md`는 새 팀원을 위한 온보딩 문서이기도 합니다.

---

## 9.2 위치와 계층

`CLAUDE.md`는 세 곳에 둘 수 있습니다.
각각 범위와 용도가 다릅니다.

| 계층 | 위치 | 용도 | 공유 범위 |
|------|------|------|-----------|
| 조직 정책 | 시스템 경로* | 회사 전체 규칙 | 모든 사용자 |
| 프로젝트 | `./CLAUDE.md` | 이 프로젝트 규칙 | 소스 컨트롤 |
| 사용자 | `~/.claude/CLAUDE.md` | 내 개인 습관 | 나만 사용 |

*macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`

세 파일이 모두 존재하면 어떻게 될까요?
**전부 로드됩니다.** 계층별로 합쳐집니다.

로딩 순서는 조직 → 프로젝트 → 사용자입니다.
충돌하면 나중에 로드된 규칙이 우선합니다.

실제로 가장 많이 쓰는 것은 프로젝트 레벨입니다.
프로젝트 루트에 `CLAUDE.md`를 두거나,
`.claude/CLAUDE.md`에 두면 됩니다.

```text
my-project/
├── CLAUDE.md              # 프로젝트 규칙 (팀 공유)
├── .claude/
│   └── CLAUDE.md          # 이쪽에 둬도 됩니다
├── src/
└── package.json
```

> **팁**: 팀과 공유할 규칙은 프로젝트 `CLAUDE.md`에, 개인 취향(예: "응답을 한국어로 해줘")은 `~/.claude/CLAUDE.md`에 적으세요.

## 9.3 /init으로 시작하기

빈 파일부터 시작할 필요 없습니다.
`/init` 슬래시 커맨드가 자동으로 만들어줍니다.

```bash
$ cd my-project
$ claude
> /init
```

Claude Code가 프로젝트를 분석합니다.
`package.json`, `tsconfig.json`, `.eslintrc` 등
설정 파일을 읽고 규칙을 추출합니다.

```text
[Claude Code]
프로젝트를 분석하여 CLAUDE.md를 생성합니다...

✓ package.json 분석 완료
✓ tsconfig.json 분석 완료
✓ .eslintrc.js 분석 완료
✓ 디렉토리 구조 파악 완료

CLAUDE.md를 생성했습니다.
```

생성된 파일을 확인해보세요.

```markdown
# Project: my-project

## Build & Run
- Install: `npm install`
- Dev: `npm run dev`
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`

## Code Style
- TypeScript strict mode
- ESLint + Prettier
- 세미콜론 사용, 싱글 쿼트

## Project Structure
- `src/` — 소스 코드
- `src/api/` — API 라우트
- `src/components/` — React 컴포넌트
- `tests/` — 테스트 파일
```

자동 생성된 내용은 출발점입니다.
여러분의 프로젝트에 맞게 수정하세요.
"이건 빠졌네", "이건 더 구체적으로" 하면서
다듬어 나가는 것이 핵심입니다.

> **팁**: `/init`은 기존 `CLAUDE.md`가 있으면 덮어쓰지 않습니다. 처음 한 번만 사용하고, 이후에는 직접 편집하세요.

## 9.4 잘 쓰는 법

좋은 `CLAUDE.md`에는 공통점이 있습니다.

### 원칙 1: 200줄 이하로 유지하세요

`CLAUDE.md`는 매 세션마다 로드됩니다.
길어지면 컨텍스트를 낭비합니다.
핵심 규칙만 담으세요.

### 원칙 2: 구체적으로 작성하세요

추상적 지시는 효과가 없습니다.

```markdown
<!-- 나쁜 예 -->
# CLAUDE.md
- 깨끗한 코드를 작성하세요
- 베스트 프랙티스를 따르세요
- 성능을 고려하세요
```

무엇이 "깨끗한" 코드인지 모릅니다.
구체적 규칙을 주세요.

```markdown
<!-- 좋은 예 -->
# CLAUDE.md

## 빌드
- 빌드: `npm run build`
- 단일 테스트: `npm test -- --grep "테스트명"`
- 린트: `npm run lint`

## 코드 스타일
- 함수명: camelCase
- 컴포넌트명: PascalCase
- 파일명: kebab-case.ts
- 한 함수는 30줄 이내
- any 타입 사용 금지

## 테스트 규칙
- 새 함수에는 반드시 단위 테스트 추가
- 테스트 파일: `__tests__/파일명.test.ts`
- describe/it 패턴 사용

## 금지 사항
- `console.log` 커밋 금지 (디버깅 후 제거)
- `.env` 파일 수정 금지
```

### 원칙 3: 빌드/스타일/테스트에 집중하세요

Claude Code가 가장 자주 참조하는 정보입니다.
이 세 가지가 정확하면 대부분 잘 동작합니다.

> **주의**: `CLAUDE.md`에 API 키나 비밀번호를 적지 마세요. 이 파일은 소스 컨트롤에 포함될 수 있습니다.

## 9.5 고급 기능

프로젝트가 커지면 `CLAUDE.md` 하나로는
부족할 수 있습니다. 세 가지 확장 기능이 있습니다.

### @파일 임포트

외부 파일의 내용을 가져올 수 있습니다.

```markdown
# CLAUDE.md

## 공통 규칙
@docs/coding-standards.md

## API 규칙
@docs/api-guidelines.md
```

`@경로`를 만나면 해당 파일을 읽어서
`CLAUDE.md`에 합칩니다.
최대 5단계 깊이까지 중첩 임포트가 가능합니다.

이렇게 하면 `CLAUDE.md` 본체는 짧게 유지하면서
상세 규칙은 별도 파일로 관리할 수 있습니다.

### .claude/rules/ 경로별 규칙

특정 파일 경로에만 적용되는 규칙을 만들 수 있습니다.

```yaml
# .claude/rules/api-rules.md
---
paths:
  - "src/api/**/*.ts"
---
# API 파일 규칙
- 모든 엔드포인트에 에러 핸들링 필수
- 응답 형식은 { data, error, status } 통일
- Zod로 입력 검증 필수
```

```yaml
# .claude/rules/test-rules.md
---
paths:
  - "**/*.test.ts"
  - "**/*.spec.ts"
---
# 테스트 파일 규칙
- AAA 패턴 (Arrange-Act-Assert) 사용
- 모킹은 vi.mock() 사용
- 각 테스트는 독립적으로 실행 가능해야 함
```

Claude Code가 `src/api/users.ts`를 편집할 때
API 규칙이 자동으로 적용됩니다.
테스트 파일을 편집할 때는 테스트 규칙이 적용됩니다.

규칙이 필요한 파일에만 로드되므로
컨텍스트를 절약할 수 있습니다.

### Auto Memory

Claude Code는 대화 중 학습한 내용을
자동으로 기억할 수 있습니다.

"앞으로 커밋 메시지는 한글로 작성해줘"라고 하면
이 선호사항이 메모리에 저장됩니다.
다음 세션에서도 기억합니다.

저장 위치는 `~/.claude/projects/<project>/memory/`입니다.
`MEMORY.md` 파일의 처음 200줄이 매 세션 로드됩니다.

> **팁**: Auto Memory는 편리하지만, 중요한 규칙은 `CLAUDE.md`에 명시적으로 적는 것이 더 확실합니다. 메모리는 보조 수단으로 활용하세요.

## 9.6 안티패턴

마지막으로, 피해야 할 패턴을 정리합니다.

### 안티패턴 1: 300줄 넘는 거대한 파일

모든 것을 `CLAUDE.md`에 넣으려 하지 마세요.
컨텍스트를 낭비하고, 규칙끼리 충돌합니다.

**해결**: `@임포트`와 `.claude/rules/`로 분산하세요.

### 안티패턴 2: 추상적이고 모호한 지시

"좋은 코드를 작성하세요"는 아무 효과가 없습니다.
"함수는 30줄 이내, any 금지"가 효과적입니다.

**해결**: 검증 가능한 구체적 규칙을 작성하세요.

### 안티패턴 3: 한 번 쓰고 방치

프로젝트는 진화합니다.
`CLAUDE.md`도 함께 업데이트해야 합니다.

**해결**: 코드 리뷰 때 `CLAUDE.md`도 함께 검토하세요.

### 안티패턴 4: 중복 규칙

같은 규칙이 `CLAUDE.md`와 `.eslintrc`에
모두 있으면 혼란을 줍니다.

**해결**: "린트 규칙은 `.eslintrc.js`를 따르세요"로
위임하는 것이 깔끔합니다.

## 정리

| 핵심 포인트 | 설명 |
|------------|------|
| 역할 | 매 세션 자동 로드되는 프로젝트 규칙 |
| 계층 | 조직 → 프로젝트 → 사용자 (3단계) |
| 시작 | `/init`으로 자동 생성 후 수정 |
| 분량 | 200줄 이하, 빌드/스타일/테스트 중심 |

| 금지 | 300줄+ 파일, 추상적 지시, 방치 |

`CLAUDE.md`는 여러분과 Claude Code 사이의 계약서입니다.
명확하게 쓸수록, Claude Code는 더 정확하게 일합니다.
다음 챕터에서는 권한과 보안 설정을 살펴보겠습니다.
